## 🧠 My Personalized Claude Strategy

### ✅ Recommended Primary Model: `Claude Sonnet 4.6`

> The everyday coding powerhouse — smart, fast, and built for deep technical work.

As a daily coding student, I need a model that understands complex logic, debugs real errors, explains concepts clearly, and handles long files — without burning through usage limits. **Sonnet** hits that sweet spot perfectly.

---

## 🔀 Model Selection Guide

### ⚡ When to Use Haiku
- Quick syntax checks
- Simple one-liner questions
- Checking what a function does
- Fast lookups — no deep reasoning needed

### 🎯 When to Use Sonnet *(Daily Driver)*
- Debugging errors and exceptions
- Writing functions and features
- Understanding concepts (OOP, recursion, algorithms)
- Code explanations and walkthroughs
- Day-to-day project work

### 🧠 When to Use Opus
- System architecture design
- Hard algorithm challenges
- Capstone project planning
- Interview / placement preparation
- Major high-stakes code reviews

---

## ⚙️ Effort Level Guide

| Level | When to Use |
|-------|------------|
| **Low** | Quick syntax help, short snippets, yes/no answers |
| **Standard** ✅ *(default)* | Debugging, explaining concepts, building features |
| **High** | Multi-file refactoring, complex bug hunts, DSA problems |
| **Max** | Major assignments, capstone projects, interview prep |

## 🚫 Biggest Mistakes to Avoid

1. **Using Opus for every task** — It's slow and overkill for daily coding. Save it for architecture or interviews only.
2. **Pasting code without context** — Always explain what the code is supposed to do and what error you're seeing.
3. **Accepting code without understanding it** — Ask Claude to explain every line. You're a student; learning beats copy-pasting.
4. **Using Max effort for trivial tasks** — Wastes time and context. Match effort to task complexity.
5. **One-shot giant prompts** — Break big problems into steps. "Build me a full app" gets worse results than iterative building.

---

## 🌟 Final Recommendation

> **If I could only pick one model and one effort level:**
>
> ### 🎯 Claude Sonnet 4.6 · Standard Effort
>
> It handles 90% of student coding tasks — debugging, learning, building features — with the best balance of speed, depth, and reliability. Switch to **High** when stuck on hard problems, and pull out **Opus** only for the most important work like capstone projects or placement interviews.

## 💡 My Optimal Daily Coding Workflow

```
1. Give Claude full context at the start of each session
   → Language, goal, constraints, what I've already tried

2. Iterate in small steps
   → Don't ask for everything at once

3. When hitting a bug:
   → Paste the error message + relevant code together

4. After getting a fix:
   → Always ask "Can you explain why this works?"

5. For big projects:
   → Use Claude Code (terminal/VS Code extension) for file-aware help
```

## 📚 Key Learnings

- Not all Claude models are equal — choosing the right one saves time and improves output quality.
- **Effort level** matters as much as model choice; over-powering simple tasks is wasteful.
- As a student, the most important habit is **asking Claude to explain**, not just provide answers.
- Breaking problems into small iterative steps produces far better code than one giant prompt.
- **Claude Code** (CLI / VS Code) is a game-changer for real project work vs. copy-pasting in chat.
